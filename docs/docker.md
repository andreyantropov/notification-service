# Docker

Сервис упаковывается в минимальный образ на базе `node:24.14.0-alpine`.
Конфигурация передаётся **только через переменные окружения**. Файлы конфигов внутри образа отсутствуют.

## 1. Сборка образа

Выполняется из корня проекта (где лежит `Dockerfile`).

```bash
docker build -t notification-service:1.0.0 .
```

**Если сборка идёт за прокси:**

```bash
docker build \
  --build-arg HTTP_PROXY=$HTTP_PROXY \
  --build-arg HTTPS_PROXY=$HTTPS_PROXY \
  -t notification-service:1.0.0 .
```

> **Примечание:** В `Dockerfile` уже настроен кэш зависимостей и удаление dev-пакетов. Лишние слои не создаются.

## 2. Запуск

Сервис требует наличия файла `.env` с настройками каналов доставки (SMTP, Bitrix и т.д.).

```bash
docker run -d \
  --name notification-service \
  --restart unless-stopped \
  --env-file .env \
  -p 3000:3000 \
  notification-service:1.0.0
```

- **Порт:** Внутри контейнера по умолчанию используется `3000`. При изменении `SERVICE_PORT` в `.env` не забудьте обновить публикацию порта в команде `docker run` (например, `-p 4000:4000`).
- **Пользователь:** Запускается от имени `node` (без прав root).
- **Сигналы:** Использует `dumb-init` для корректной обработки остановки (`SIGTERM`).

**Посмотреть логи запуска**

`docker logs -f notification-service`

**Проверить состояние контейнера**

`docker ps -f name=notification-service`

## 3. Требования к окружению

Перед запуском убедитесь, что из контейнера есть сетевой доступ к внешним сервисам, указанным в `.env`:

- **SMTP сервер** (для почты)
- **Bitrix24 API** (если используется канал)
- **OpenTelemetry Collector** (если включена телеметрия)
