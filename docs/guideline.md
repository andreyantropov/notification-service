# 🧑‍💻 Гайдлайн по стилю и неймингу

> Этот документ дополняет автоматические проверки (ESLint, Prettier, TypeScript), покрывая зоны, где важна человеческая договорённость.  
> Цель: единообразие, предсказуемость и читаемость без избыточности.

Проект разрабатывается в **функциональном стиле**:

- компоненты реализуются как **чистые фабричные функции**,
- **нет классов**, глобального состояния или мутаций,
- зависимости передаются явно, через параметры,
- поведение расширяется через **композицию и декораторы**, а не наследование.

Мы придерживаемся фундаментальных принципов:

- **KISS** — «Keep It Simple, Stupid»: простота важнее сложной гибкости,
- **DRY** — «Don’t Repeat Yourself»: дублирование допустимо только при отсутствии общей семантики,
- **YAGNI** — «You Aren’t Gonna Need It»: не добавляй функциональность «на будущее».

---

## 📦 Импорты

- Все импорты **обязаны** содержать расширение `.js`:  
  `import { X } from './x.js';`
- Импорты из других модулей — **только через barrel-файлы**:  
  `import { X } from '../domain/index.js';`
- Не импортируй **напрямую** файлы за пределами текущей директории.
- Для импорта/экспорта типов используй инлайн-синтаксис `import { type X }` и `export { type X }`, а не `import type { X }` / `export type { X }`.

---

## 🏷️ Нейминг

- Переменные, функции, параметры — `camelCase`.
- Типы, интерфейсы, классы — `PascalCase`.
- Константы (ключи, порты, таймауты, настройки) — `UPPER_CASE`.
- Избегай избыточных слов в именах, если контекст и так ясен (`NotificationSender` → `Sender`).
- Имена файлов — `camelCase`.

---

## 🗃️ Структура проекта

- Структура следует принципам **чистой архитектуры**:  
  `domain` → `application` → `infrastructure` → `presentation`, с `shared` как общим вспомогательным слоем.
- **Во всех директориях, кроме корневых архитектурных (`domain`, `application` и т.д.)**, обязателен `index.ts` (barrel-файл).
- Все публичные сущности экспортируются **только через barrel-файлы**.
- Интерфейсы, константы и типы выносятся в отдельные файлы.

---

## 🧱 Фабрики и декораторы

- Фабричные функции именуются как `createXxx`.
- Декораторы именуются как `withXxx`.
- Принимают **два параметра**: `dependencies` (инъекции) и `config` (настройки).
- Все параметры — объекты, даже если один.
- Возвращаемый объект содержит только публичный API.
- Имя файла должно совпадать с именем основной экспортируемой сущности.

---

## 🧩 Типизация

- Все функции, параметры и возвращаемые значения — **явно типизированы**.
- Объявляй функции как `const` с явной типизацией, а не через `function`.
- Union-типы — через `type`; объектные структуры — через `interface`.
- Использование `any` **запрещено** (гарантируется настройкой `tsconfig.json`).
- Для неизменяемых данных — используй `readonly`.
- Generic-типы должны быть строго ограничены; избегай `unknown` без последующей проверки.
- Используй `readonly` для полей интерфейсов и свойств объектов, которые не должны изменяться после инициализации (конфигурации, зависимости, доменные значения).

---

## 🛠 Очищение кода

- Удаляй **все комментарии**, включая `// TODO` и `// FIXME`. Код должен быть самодокументируемым.
- Удаляй мёртвый и неиспользуемый код.
- **Телеметрия не должна быть захардкожена.**  
  Она применяется **через декораторы**, которые получают зависимости (`Logger`, `Meter`, `Tracer`) через инъекцию.
- Все имена метрик и атрибутов для OpenTelemetry — в `snake_case` (преобразуются утилитами при экспорте).

---

## 🧪 Тесты

- Все зависимости мокаются через DI.
- Тесты зависят только от публичного API, а не от реализации.
- Используй единый стиль описания (`describe`, `it`).
- Покрывай как успешные сценарии, так и ошибочные.

---

> 💡 **Совет**: если код выглядит «не так, как в соседних файлах» — скорее всего, он нарушает гайдлайн. Сверяйся с живыми примерами.  
> При сомнениях — открой обсуждение, не коммить в обход стандарта.
