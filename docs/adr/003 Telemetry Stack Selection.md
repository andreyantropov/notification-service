# ADR 003: Выбор стека для сбора и хранения телеметрии (логи, трейсы, метрики)

**Дата:** 2025-11-13  
**Участники:**

...

## Статус

Принято  
_2025-11-03_

## Контекст

В текущей архитектуре сервис использует OpenTelemetry SDK для генерации логов, трейсов и метрик. Логи собираются через `winston` с `OpenTelemetryTransportV3`, трейсы — через `BatchSpanProcessor`, метрики — через стандартные инструменты Node.js. Все данные экспортируются в консоль при разработке, но для production требуется централизованный стек телеметрии.

Цель: выбрать оптимальные инструменты для сбора, хранения и визуализации логов, трейсов и метрик с учётом:

- Наличия уже развернутой Grafana (для визуализации)
- Ограниченных серверных ресурсов
- Малого количества сервисов
- Требования к простоте развертывания и поддержки
- Совместимости с OpenTelemetry (OTLP)
- Минимизации кастомной логики и технического долга

Рассматривались варианты:

1. **OTel Collector** как централизованный агент сбора
2. **Grafana Agent + Promtail** как альтернатива

Для хранения:

- **Логи:** Loki (выбран как единственный разумный вариант)
- **Трейсы:** Tempo vs. Jaeger
- **Метрики:** Prometheus vs. Mimir

## Варианты решения

### 1. OTel Collector

**✅ Преимущества:**

- Нативная поддержка OTLP — идеально совместим с OpenTelemetry SDK, используемым в проекте.
- Единая точка сбора для логов, трейсов и метрик — упрощает конфигурацию и мониторинг.
- Поддержка фильтрации, преобразования и агрегации данных до отправки (например, удаление чувствительных полей, переформатирование).
- Поддерживает экспорт в Loki, Tempo, Prometheus — позволяет гибко настраивать целевые системы.
- Легко масштабируется: можно развернуть как sidecar или централизованно.
- Не требует изменения кода сервисов — достаточно настроить экспортер на OTLP endpoint.
- Поддерживает batch-обработку и retry-механизмы, повышает надёжность.

**❌ Недостатки:**

- Конфигурация может стать сложной при росте числа правил.
- Требует небольшого обучения команды для настройки pipeline'ов.

### 2. Grafana Agent + Promtail

**✅ Преимущества:**

- Интеграция с Loki и Grafana "из коробки".
- Легковесен, хорошо работает с файловыми логами.

**❌ Недостатки:**

- **Promtail не умеет корректно собирать OTLP-логи** — он предназначен только для чтения файлов (например, `*.log`), а не приёмки OTLP-потоков.
- Нет нативной поддержки трейсов и метрик в одном агенте — потребуется отдельный агент для метрик (Grafana Agent — metrics) и отдельный для трейсов (не поддерживается).
- Невозможно использовать единый пайплайн для всех типов телеметрии — усложняет архитектуру.
- Не поддерживает преобразование OTLP-событий (например, извлечение traceId для логов) — теряется связность данных.
- Для трейсов и метрик всё равно потребуется другой инструмент — не решает задачу централизации.
- **Итог: не подходит как единое решение для полного стека OpenTelemetry.**

## Решение

**Выбранный вариант:** **OTel Collector** как единый сборщик телеметрии.

**Причина выбора:**  
OTel Collector — единственный инструмент, который **натурально** интегрируется с нашим текущим стеком OpenTelemetry и позволяет собрать **все три потока телеметрии (логи, трейсы, метрики)** в одной точке, с минимальными изменениями в сервисах.

## Хранение данных

#### 🔹 Логи: **Loki**

**Loki**

- ✅ **Оптимизирован под логи**: хранит только сами логи и метаданные (labels), не индексирует полнотекстово — это резко снижает потребление диска и памяти.
- ✅ **Глубокая интеграция с Grafana**: нативный datasource, поддержка трассировки через `trace_id`, возможность корреляции с метриками и трейсами.
- ✅ **Label-based фильтрация**: быстрый поиск по `service.name`, `level`, `span_id` и другим структурированным полям.
- ✅ **Масштабируемость через object storage**: данные можно хранить в S3/MinIO, что упрощает долгосрочное хранение и отказоустойчивость.
- ❌ **Не предназначен для full-text поиска**: если нужен сложный анализ текста.
- ❌ **Требует структурированных логов**: эффективность зависит от качества логирования (наличие traceId, service.name и т.п.).

👉 **Вывод:** В условиях уже развернутой Grafana, ограниченных ресурсов и необходимости именно централизованного просмотра логов — **Loki является оптимальным выбором**. Его недостатки не критичны, а преимущества полностью соответствуют требованиям.

✅ **Выбрано: Loki**

---

#### 🔹 Трейсы: **Tempo vs Jaeger**

**Tempo**

- ✅ **Полная интеграция с экосистемой Grafana**: один UI, единая аутентификация, корреляция с логами и метриками.
- ✅ **Простота развертывания и эксплуатации**: минимальная конфигурация, легко запустить как monolithic или microservices режим.
- ✅ **Низкое потребление ресурсов**: идеально подходит для средних и малых нагрузок.
- ✅ **Поддержка OTLP "из коробки"**: прямая совместимость с OpenTelemetry Collector.
- ✅ **Хранение в object storage (S3/MinIO)**: упрощает резервное копирование и масштабирование.
- ❌ **Ограниченный функционал анализа зависимостей**: нет встроенных service maps или dependency graphs (но можно интегрировать с внешними инструментами).
- ❌ **Меньше активности в сообществе**, чем у Jaeger (но поддерживается Grafana Labs).

**Jaeger**

- ✅ **Богатый функционал**: service maps, dependency graphs, advanced search, sampling strategies.
- ✅ **Зрелое решение** с большим комьюнити и документацией.
- ❌ **Высокое потребление ресурсов**: особенно в production-кластерах (query, collector, ingester).
- ❌ **Сложнее в настройке и поддержке**: требует бэкенд для хранения (Cassandra, Elasticsearch) и отдельной настройки масштабирования.
- ❌ **Плохая интеграция с Grafana**: хотя есть datasource, он менее удобен, чем Tempo.

👉 **Вывод:** Несмотря на функциональное превосходство Jaeger, его сложность и ресурсоёмкость **не оправданы** при нашем объёме (5–7 сервисов, низкая нагрузка). Tempo проще, легче, лучше интегрируется с Grafana и **полностью покрывает наши потребности**.

✅ **Выбрано: Tempo**

---

#### 🔹 Метрики: **Prometheus vs Mimir**

**Prometheus**

- ✅ **Простота и надёжность**: single-binary решение, легко развернуть и мониторить.
- ✅ **Отличная интеграция с Grafana**: нативный datasource, поддержка alerting, recording rules, dashboard'ов.
- ✅ **Поддержка OTLP через OpenTelemetry Collector**: может принимать метрики напрямую от Collector.
- ✅ **Идеален для pull-модели**: подходит для большинства наших сервисов (Node.js, Express).
- ✅ **Гибкость в настройке SLO/SLI**: мощный query язык (PromQL), простые правила алертинга.
- ❌ **Ограниченное долгосрочное хранение**: обычно 15–30 дней без remote storage.
- ❌ **Сложнее горизонтальное масштабирование**.

**Mimir**

- ✅ **Горизонтально масштабируемый**: подходит для крупных систем с тысячами метрик.
- ✅ **Долгосрочное хранение и высокая доступность**: enterprise-grade решение.
- ✅ **Pull и push модели**: гибкость в сборе данных.
- ❌ **Высокая сложность**: требует Kubernetes, etcd, объектное хранилище, много компонентов.
- ❌ **Избыточен для нашего масштаба**: мы не достигаем даже 1% от порога, где Mimir становится оправданным.
- ❌ **Увеличивает технический долг**: сложнее поддерживать, больше точек отказа.

👉 **Вывод:** Mimir — отличный выбор для large-scale систем, но **является overkill** для текущего окружения. Prometheus — проверен, прост, эффективен и **полностью соответствует нашим требованиям**. При росте — можно добавить `remote_write` в Mimir или Thanos.

✅ **Выбрано: Prometheus**

| Данные      | Рассматривались   | **Выбранный вариант** | Обоснование                                                                     |
| ----------- | ----------------- | --------------------- | ------------------------------------------------------------------------------- |
| **Логи**    | Loki              | ✅ **Loki**           | Интеграция с Grafana, низкие требования к ресурсам, label-based search          |

## Последствия

### Положительные

- **Единый и простой стек**: OTel Collector → Loki (логи), Tempo (трейсы), Prometheus (метрики).
- **Полная совместимость с OpenTelemetry** — не требует кастомных экспортеров.
- **Единый UI в Grafana** — логи, метрики и трейсы доступны в одном месте с возможностью перехода по traceId.
- **Минимальные ресурсные затраты** — все компоненты легко размещаются на одном сервере.
- **Низкий технический долг** — нет кастомных решений, всё — стандартные инструменты.
- **Готовая инфраструктура для SLO/SLI** — Prometheus + Grafana позволяют легко задать и визуализировать метрики надёжности.

### Отрицательные

- Не поддерживает batch-обработку трейсов/логов на уровне Collector — но это не критично при низкой нагрузке.
- Требуется настройка pipeline'ов в Collector (но конфигурация проста и шаблонизирована).

## Риски и меры их смягчения

| Риск                                    | Вероятность  | Влияние | Меры по смягчению                                                                                                                |
| --------------------------------------- | ------------ | ------- | -------------------------------------------------------------------------------------------------------------------------------- |
| OTel Collector становится точкой отказа | Низкая       | Среднее | Развернуть в режиме HA (2 инстанса) с балансировкой; мониторинг через health endpoint и alerting в Grafana                       |
| Сложность конфигурации Collector        | Низкая       | Низкое  | Использовать готовые шаблоны из OpenTelemetry Helm charts / Docker Compose; документировать конфиг как часть инфраструктуры кода |
| Задержки в сборе логов/трейсов          | Очень низкая | Низкое  | Настроить `batch`-обработку в экспортерах клиентов (уже сделано); использовать `timeout` и `retry` в Collector                   |
| Недостаток ресурсов на сервере          | Низкая       | Низкое  | Мониторинг потребления CPU/RAM через Prometheus; настройка лимитов в Docker/K8s; при росте — перенос на отдельный узел           |
