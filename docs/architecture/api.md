# API

Этот документ описывает HTTP API сервиса уведомлений.  
Полная спецификация доступна в интерактивном виде по адресу `/api-docs/` (Swagger UI).

---

## Аутентификация

Все бизнес-эндпоинты (`/notifications/*`) защищены **JWT-аутентификацией**.  
Клиенты обязаны передавать валидный токен (Keycloak) в заголовке:

```http
Authorization: Bearer <JWT>
```

Эндпоинты `/health/*` и `/api-docs/` — **публичные**, аутентификация не требуется.

---

```
{{baseUrl}}/api
```

## 1. Отправка одиночного уведомления

**Endpoint:** `POST /notifications`

Принимает одно уведомление для доставки одному получателю через один или несколько каналов.

### Структура запроса

```json
{
  "contacts": [
    { "type": "email", "value": "user@company.com" },
    { "type": "bitrix", "value": 12345 }
  ],
  "message": "Приветик через Интернетик!"
}
```

#### Параметры тела запроса

| Поле       | Тип    | Обязательное | Описание                                                              |
| :--------- | :----- | :----------- | :-------------------------------------------------------------------- |
| `contacts` | Array  | **Да**       | Список контактов получателя. Минимум 1, максимум 50. См. формат ниже. |
| `message`  | String | **Да**       | Текст уведомления (1–10 000 символов).                                |
| `strategy` | String | Нет          | Стратегия доставки. По умолчанию: `send_to_first_available`.          |

#### Формат контакта (`contacts[]`)

Используется дискриминируемый союз (`discriminatedUnion`):

- **Email:** `{ "type": "email", "value": "user@example.com" }`
- **Bitrix24:** `{ "type": "bitrix", "value": 12345 }` (числовой ID пользователя)

#### Стратегии доставки (`strategy`)

- `send_to_first_available` — отправка прекращается после успеха в первом канале.
- `send_to_any_available` — попытка отправки во все каналы. Успех, если доставлен хотя бы один; ошибка, если все упали.
- `send_to_all_available` — успех, если уведомление доставлено **по всем** доступным каналам; ошибка, если хотя бы по одному каналу отправить уведомление не удалось.

> ⚠️ Поля `id`, `createdAt`, `initiator` генерируются сервером и игнорируются во входных данных.

### Ответы

| Код                         | Описание                                                                            |
| :-------------------------- | :---------------------------------------------------------------------------------- |
| `200 OK`                    | Уведомление успешно принято в обработку. Возвращается полный объект `Notification`. |
| `400 Bad Request`           | Ошибка валидации (пустые контакты, нет сообщения).                                  |
| `401 Unauthorized`          | JWT токен отсутствует или невалиден.                                                |
| `403 Forbidden`             | Пользователь не имеет прав на отправку.                                             |
| `500 Internal Server Error` | Внутренняя ошибка сервиса.                                                          |

#### Пример успешного ответа (`200`)

```json
{
  "id": "a1b2c3d4-e5f6-7890-g1h2-i3j4k5l6m7n8",
  "createdAt": "2026-03-09T10:00:00.000Z",
  "contacts": [{ "type": "email", "value": "user@company.com" }],
  "message": "Ваш заказ №12345 готов к выдаче",
  "strategy": "send_to_first_available",
  "initiator": {
    "id": "user-123",
    "name": "John Doe"
  }
}
```

## 2. Пакетная отправка уведомлений

**Endpoint:** `POST /notifications/batch`

Принимает массив уведомлений (от 1 до 50) для массовой рассылки.

### Структура запроса

Тело запроса — **массив** объектов, идентичных запросу одиночной отправки.

```json
[
  {
    "contacts": [{ "type": "email", "value": "admin@company.com" }],
    "message": "Отчет сформирован",
    "strategy": "send_to_all_available"
  },
  {
    "contacts": [{ "type": "bitrix", "value": 999 }],
    "message": "Требуется согласование"
  }
]
```

### Принцип частичной валидации

Сервис валидирует каждое уведомление в массиве независимо:

- ✅ **Валидные** уведомления принимаются в работу и получают `id`.
- ❌ **Невалидные** отклоняются с описанием ошибки, но **не блокируют** обработку остальных.

> 💡 Это позволяет избежать ситуации, когда одна опечатка в пакете из 50 сообщений отменяет доставку всех 49 корректных.

> 🔄 Гарантия порядка: Массив результатов в ответе (`payload`) всегда строго соответствует порядку элементов в запросе.
>
> - Элемент `payload[0]` — это результат обработки первого уведомления из запроса.
> - Элемент `payload[N]` — результат N-ного уведомления.
>   Это позволяет корректно сопоставлять ошибки валидации с исходными данными даже для элементов, которые не получили `id`.

### Коды ответов

| Код                | Описание                                                               |
| :----------------- | :--------------------------------------------------------------------- |
| `202 Accepted`     | **Все** уведомления в пакете валидны и приняты.                        |
| `207 Multi-Status` | **Частичный успех**: часть уведомлений принята, часть отклонена.       |
| `400 Bad Request`  | Нарушена структура пакета (например, пустой массив или >50 элементов). |
| `401 Unauthorized` | JWT токен отсутствует или невалиден.                                   |
| `403 Forbidden`    | Пользователь не имеет прав на отправку.                                |
| `500`              | Внутренняя ошибка.                                                     |

#### Пример ответа `207 Multi-Status`

```json
{
  "message": "Уведомления приняты частично",
  "summary": {
    "total": 2,
    "accepted": 1,
    "rejected": 1
  },
  "payload": [
    {
      "status": "success",
      "payload": {
        "id": "uuid-1...",
        "createdAt": "2026-03-09T10:00:00.000Z",
        "contacts": [{ "type": "email", "value": "admin@company.com" }],
        "message": "Отчет сформирован",
        "strategy": "send_to_all_available",
        "initiator": { "id": "user-123", "name": "John Doe" }
      }
    },
    {
      "status": "client_error",
      "error": {
        "message": "Список контактов не должен быть пустым",
        "details": {
          "code": "too_small",
          "path": ["contacts"]
        }
      }
    }
  ]
}
```

## Healthcheck Эндпоинты

Используются оркестраторами (Kubernetes) для контроля состояния сервиса.

| Endpoint            | Тип       | Описание                                                                  |
| :------------------ | :-------- | :------------------------------------------------------------------------ |
| `GET /health/live`  | Liveness  | Процесс запущен и отвечает. Если падает — контейнер перезапускается.      |
| `GET /health/ready` | Readiness | Сервис готов принимать трафик (все зависимости: SMTP, Bitrix — доступны). |

## Рекомендации для интеграторов

1.  **Используйте пакетную отправку** для массовых рассылок (до 50 шт. за запрос) — это снижает нагрузку на сеть и сервис.
2.  **Не повторяйте запрос при `400` (или элементы 207 со статусом `CLIENT_ERROR`)** без исправления данных — ошибка валидации не решится повтором.
3.  **Повторяйте запрос при `5xx`** с экспоненциальной задержкой.
4.  **Сохраняйте `id` уведомления** из ответа для корреляции с логами и трейсами.
