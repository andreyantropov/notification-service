# 🏗️ infra/ (Logical Configuration)

Директория содержит **критические логические конфигурации инфраструктуры**, неразрывно связанные с бизнес-логикой сервиса. Здесь описаны статические контракты взаимодействия с внешними системами: топология очередей сообщений, политики безопасности и правила маршрутизации на уровне шлюза.

Эти файлы являются частью кодовой базы и версионируются вместе с приложением, так как их изменение требует синхронного обновления кода сервисов (например, изменение имени очереди или права доступа требует деплоя новой версии приложения).

> **Note:** Динамические настройки окружения (ресурсы CPU/RAM, сетевые политики Kubernetes, секреты) вынесены в отдельный репозиторий инфраструктуры. В этой папке хранятся только **логические** настройки, определяющие _как_ работает система, а не _где_ она запущена.

---

## 📂 Структура и назначение

Директория организована по типу внешних зависимостей. Каждый подкаталог содержит декларативное описание контракта между сервисом и инфраструктурным компонентом.

### 1. `rabbitmq/` — Топология сообщений

Определяет логическую структуру брокера очередей, необходимую для гарантированной доставки уведомлений.

- **Назначение**: Фиксация имен сущностей (Exchanges, Queues, Bindings), стратегий обработки ошибок (Dead Lettering), параметров надежности (durability) и глобальных политик управления очередями (Policies).
- **Ключевой принцип**: Полная декларативность. Вся топология (включая политики) создается автоматически при старте инфраструктуры, исключая ручное создание через UI, человеческий фактор и риск рассинхронизации конфигураций между средами.

### 2. `nginx/` – Правила входа (Ingress)

Описывает политику доступа к API сервиса на уровне шлюза.

- **Назначение**: Определение эндпоинтов, правил ограничения частоты запросов (Rate Limiting) и таймаутов соединения.
- **Ключевой принцип**: Конфиг параметризован переменными окружения для адаптации под разные нагрузки (dev/prod), но сама структура маршрутов остается неизменной частью контракта API.

### 3. `keycloak/` – Модель доступа (IAM)

Содержит шаблон конфигурации Realm, определяющий права доступа и клиентов для M2M-взаимодействия.

- **Назначение**: Описание ролевой модели (RBAC) и настроек клиентов (Service Accounts), используемых для аутентификации внутренних сервисов.
- **Ключевой принцип**: Чувствительные данные вынесены в переменные окружения, тогда как структура ролей и политиков версионируется как код.

## 🛠 Принципы работы

### 1. Синхронизация с кодом

Любое изменение в этих файлах (например, переименование очереди или роли) считается **breaking change** для инфраструктуры и должно сопровождаться обновлением констант в коде сервиса.

### 2. Параметризация

Файлы используют синтаксис переменных окружения (например, `${VAR}`) для значений, зависящих от среды.

- **Локально**: Значения подставляются через `.env` файл при запуске Docker Compose.
- **В CI/CD**: Значения инжектируются через Helm values или Kustomize перед применением конфигурации.

### 3. Идемпотентность развертывания

Конфигурации спроектированы так, чтобы их повторное применение не приводило к ошибкам (например, создание уже существующей очереди в RabbitMQ игнорируется, если параметры совпадают).
